Контекст проекта: Habits — трекер привычек с системой жетонов

Что это за продукт

Веб-приложение (SPA, без SSR) для самодисциплины через токен-экономику.
Пользователь выполняет привычки → получает жетоны → тратит жетоны на
самостоятельно назначенные награды. НЕ ИИ-коуч, никакой диагностики —
вся логика детерминирована и определяется пользователем в рамках заданных
пресетов (чтобы исключить самообман/накрутку).

Стек

Frontend:


Vite + React + TypeScript
React Router — навигация
TanStack Query — работа с API, кэш, оптимистичные обновления
Zustand — глобальный стейт (текущий юзер, UI-состояния)
Tailwind CSS — стилизация
React Hook Form + Zod — формы и валидация


Backend:


NestJS (TypeScript)
Prisma (или прямой SQL-клиент) поверх Supabase Postgres
ВСЯ бизнес-логика и расчёты жетонов — только на бэкенде. Фронт никогда
не пишет напрямую в Supabase (никакого Supabase client SDK для записи
бизнес-данных с фронта). Фронт общается только с NestJS API.


Инфраструктура:


Supabase — только как managed Postgres + Auth (email/password, возможно
OAuth). Storage — по необходимости.


Бизнес-логика (критично для реализации)

Типы привычек


instant — разовая задача. Выполнение → начисление жетонов по пресету
сложности → привычка удаляется из активных, запись переносится в историю
со статусом completed.
conditional — челлендж на N дней подряд (required_days, задаёт юзер
при создании). Ежедневные отметки идут в habit_logs. Стрик считается
ТОЛЬКО непрерывным от старта:

Если дошёл до N дней подряд → полные жетоны по пресету, статус completed.
Если прервал раньше (пропустил день) → жетоны начисляются по формуле:
итог = round(жетоны_за_привычку * (выполненные_дни_подряд / required_days))
Статус partial. Привычка после этого закрывается (удаляется из активных).



recurring — повторяющаяся привычка по расписанию (schedule: daily или
конкретные дни недели). Выполнение → жетоны сразу по пресету → привычка
НЕ удаляется, просто ждёт следующего запланированного дня.


Пресеты сложности (жетоны фиксированы, свободный ввод суммы запрещён)


Лёгкая: 5
Средняя: 10
Сложная: 20
(Точные значения могут быть вынесены в конфиг/константы, но принцип —
никогда не давать пользователю вводить произвольное число жетонов за
выполнение привычки.)


FULL-бонус

Если за календарный день выполнены ВСЕ привычки, у которых было запланировано
выполнение на этот день (recurring по расписанию + активные instant/conditional
с отметкой на сегодня) — начисляется дополнительный фиксированный бонус.
Бонус должен начисляться идемпотентно — не задваиваться при повторных
чеках/анчеках в течение дня. Бонус на выходных может быть выше буднего
(значение — вынести в конфиг).

Награды (Wishlist)


Пользователь создаёт награду: название + цена в жетонах.
Минимальный порог цены — ФИКСИРОВАННОЕ значение 100 жетонов, захардкожено
в конфиге бэкенда. НЕ настраивается пользователем ни в каком виде (иначе
юзер просто занизит порог себе сам, и вся защита от самообмана теряет
смысл). Валидация строго на бэкенде, с понятной ошибкой при попытке
создать награду дешевле 100.
Максимума нет.


Redeem (трата жетонов)


Юзер отмечает, что реально потратил жетоны на награду IRL.
Записывается в redeems: дата, сумма списания, ссылка на награду.
Баланс уменьшается.


Баланс жетонов


НЕ хранить как одно кэшируемое число без источника правды. Рекомендуется
вести wallet_transactions (append-only ledger: +жетоны за привычку,
+бонус FULL, -жетоны за redeem) и считать баланс как SUM() по этой таблице
(либо кэшировать агрегат, но источник правды — леджер).


Схема данных (концептуальная, адаптируй под Prisma/SQL)

habits (активные)
  id, user_id, name, type (instant|conditional|recurring)
  difficulty (easy|medium|hard) -> определяет жетоны
  required_days (nullable, только для conditional)
  schedule (nullable, только для recurring: daily | day_of_week[])
  created_at, status

habit_logs (ежедневные отметки — для recurring и для трекинга стрика conditional)
  id, habit_id, date, completed (bool)

habit_history (архив, ИСТОЧНИК ПРАВДЫ для аналитики, не зависит от habits)
  id, user_id
  habit_id UUID NULL REFERENCES habits(id) ON DELETE SET NULL  -- необязательная ссылка
  habit_name TEXT NOT NULL       -- снапшот на момент завершения
  habit_type TEXT NOT NULL       -- снапшот типа
  status (completed|not_completed|partial)
  days_completed, days_required (nullable, для conditional)
  coins_awarded
  started_at, ended_at

rewards (wishlist)
  id, user_id, name, cost_coins

redeems
  id, user_id, reward_id, redeemed_at, amount_spent

wallet_transactions
  id, user_id, amount (+/-), source_type (habit_completion|full_bonus|redeem)
  source_id, created_at

Критичный технический момент: атомарность "сгорания" привычки

Когда instant или conditional привычка завершается (успешно или частично),
операция ДОЛЖНА быть одной транзакцией на бэкенде (не два отдельных запроса
с фронта):

sqlBEGIN;
INSERT INTO habit_history (...) SELECT ... FROM habits WHERE id = $1;
INSERT INTO wallet_transactions (...) VALUES (...);
DELETE FROM habits WHERE id = $1;
COMMIT;

Вся эта логика — в одном сервисном методе NestJS (например
HabitsService.completeHabit()), никогда не размазывать между
фронтенд-запросами.

Стиль кода и предпочтения разработчика


Чистый модульный код, без избыточных абстракций
Прагматичные архитектурные решения вместо переусложнения
TypeScript строгий (strict mode)
Формулы/константы (пресеты жетонов, пороги, бонусы) — выносить в конфиг,
не хардкодить в бизнес-логике


Что НЕ входит в MVP (сознательно отложено)


RPG-слой: уровни, статы, скиллы (планируется поверх habit_history в будущем,
без переделки текущей схемы)
Мобильное приложение (текущий фокус — веб SPA)
